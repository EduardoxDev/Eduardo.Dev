# O que aprendi sendo CTO aos 15 anos

Quando co-fundei a Stackr Hosting em 2026, eu estava no segundo ano da faculdade. CTO de uma startup real, com clientes reais, pagando por disponibilidade real.

Não tem manual para isso.

## A decisão técnica mais difícil não é técnica

A primeira grande lição: as decisões que mais importam não são sobre qual linguagem usar ou qual banco de dados escolher. São sobre escopo.

O que construir primeiro? O que deixar para depois? O que nunca construir internamente?

No início, queria construir tudo do zero. Runtime de containers próprio (ok, isso foi um projeto de estudo que virou código real). Gateway de API próprio (também virou projeto). Mas queria também sistema de billing próprio, autenticação própria, monitoramento próprio...

A realidade de um time pequeno: cada coisa que você constrói tem manutenção. Para sempre.

Aprendi a distinguir entre o que é **core do negócio** (construir) e o que é **comodidade** (comprar/usar solução existente). Billing? Stripe. Auth? Keycloak. Monitoramento? Grafana + Prometheus. Infraestrutura de containers? Isso é o produto — construir.

## Disponibilidade é um SLA, não uma aspiração

Quando um serviço cai e um cliente perde dinheiro, você entende o peso da responsabilidade de uma forma que nenhum projeto acadêmico ensina.

Aprendi que disponibilidade não é um número bonito no README — é um contrato. E esse contrato exige:

- Circuit breakers em todas as dependências externas
- Graceful degradation quando componentes falham
- Runbooks escritos antes de precisar deles
- Alertas que acordam você às 3h antes do cliente reclamar

O primeiro incidente sério da Stackr durou 47 minutos. Pareceram 3 horas. Depois disso, escrevi o primeiro post-mortem da minha vida — honesto, sem culpar ninguém, focado no sistema.

## Código que você escreveu em pressa vai te assombrar

Nos primeiros meses, escrevi muito código rápido para fechar features. Funcionava. Passava nos testes. Ia pro ar.

Seis meses depois, cada feature nova exigia entender aquele código. Cada bug era mais difícil de isolar. A velocidade caiu.

A lição não é "escreva código perfeito" — é "dívida técnica tem juros". Alta taxa de juros.

Agora tenho uma regra simples: se vou precisar ler esse código de novo em 3 meses sem me lembrar por que escrevi assim, adiciono um comentário explicando o *porquê* (não o *o quê*, que o código já diz).

## A faculdade e o trabalho real

Estudo Ciência da Computação na Unifebe. É valioso — estruturas de dados, algoritmos, teoria dos grafos, fundamentos de sistemas operacionais. Coisas que uso todo dia sem perceber.

Mas tem um gap enorme entre o que a academia ensina e o que sistemas de produção exigem.

A academia ensina a implementar uma árvore B. O trabalho real exige saber *quando* usar um índice B-tree no PostgreSQL versus um índice hash — e como o planner de queries decide isso.

A academia ensina concorrência com mutex teóricos. O trabalho real exige debugar deadlocks em produção às 2h da manhã com traces incompletos.

Não é crítica à academia — é reconhecimento de que são complementares. O teórico dá o vocabulário. O prático dá o julgamento.

## O que fazer com essa dualidade

Minha estratégia: traduzir teoria em prática deliberadamente.

Aprendi sobre sistemas distribuídos na faculdade. Construí um API Gateway do zero para entender na prática o que CAP theorem significa quando Redis fica indisponível.

Aprendi sobre sistemas de arquivos. Construí um runtime de container para entender o que `pivot_root` faz no kernel quando um container sobe.

Cada projeto open source no meu GitHub é esse exercício: pegar um conceito que aprendi teoricamente e implementar até o ponto onde entendo as limitações reais.

## O que eu diria para mim de um ano atrás

Decide mais rápido. A maioria das decisões técnicas são reversíveis — o custo de decidir errado é menor que o custo de não decidir.

Documenta enquanto está fresco. O contexto que você tem enquanto constrói some em semanas.

Dorme. Código escrito às 2h da manhã invariavelmente cria problemas que precisam de 6h para resolver.

E confia mais no seu julgamento técnico. Você conhece o sistema melhor que qualquer consultor externo.
