# Spring Boot at Scale: Performance Tuning for High-Throughput APIs

Spring Boot is easy to get started with and easy to make slow at scale. This post covers the tuning techniques I've applied at Stackr Hosting to handle high-concurrency workloads without increasing hardware costs.

## 1. Connection Pool Tuning (HikariCP)

The default HikariCP configuration is too conservative for production. A pool too small causes thread starvation; too large overwhelms the database.

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 10
      connection-timeout: 3000
      idle-timeout: 600000
      max-lifetime: 1800000
      leak-detection-threshold: 10000
```

The right pool size formula: `pool_size = (core_count * 2) + effective_spindle_count`. For a 4-core machine with SSDs, start at 10 and benchmark.

Monitor pool health:

```java
@Scheduled(fixedRate = 30000)
public void logPoolStats() {
    HikariPoolMXBean pool = ((HikariDataSource) dataSource).getHikariPoolMXBean();
    log.info("Pool: active={}, idle={}, waiting={}",
        pool.getActiveConnections(),
        pool.getIdleConnections(),
        pool.getThreadsAwaitingConnection());
}
```

## 2. Async Processing with @Async

Synchronous calls to external services block threads and kill throughput under load. Move I/O-bound work off the request thread:

```java
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(8);
        executor.setMaxPoolSize(32);
        executor.setQueueCapacity(500);
        executor.setThreadNamePrefix("async-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}

@Service
public class NotificationService {

    @Async("taskExecutor")
    public CompletableFuture<Void> sendWelcomeEmail(String userId) {
        emailClient.send(userId);
        return CompletableFuture.completedFuture(null);
    }
}
```

## 3. Redis Caching with Spring Cache

Hitting the database on every request for stable data is wasteful. Cache aggressively:

```java
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))
            .serializeKeysWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new GenericJackson2JsonRedisSerializer()));

        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .build();
    }
}

@Service
public class PlanService {

    @Cacheable(value = "plans", key = "#planId")
    public Plan getPlan(String planId) {
        return planRepository.findById(planId).orElseThrow();
    }

    @CacheEvict(value = "plans", key = "#plan.id")
    public Plan updatePlan(Plan plan) {
        return planRepository.save(plan);
    }
}
```

## 4. Projections Instead of Full Entities

Fetching full JPA entities when you only need two fields is expensive — it loads all columns, maps them all, and bloats the ResultSet:

```java
public interface UserSummary {
    String getId();
    String getEmail();
}

@Repository
public interface UserRepository extends JpaRepository<User, String> {

    List<UserSummary> findByActiveTrue();

    @Query("SELECT u.id, u.email FROM User u WHERE u.createdAt > :since")
    List<UserSummary> findRecentUsers(@Param("since") Instant since);
}
```

In a table with 40 columns, this reduces data transfer by 90%+.

## 5. JVM Tuning for Low Latency

```bash
java \
  -server \
  -Xms2g -Xmx2g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=100 \
  -XX:G1HeapRegionSize=16m \
  -XX:+UseStringDeduplication \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/log/app/heap.hprof \
  -jar app.jar
```

Key decisions:
- **G1GC** — best balance of throughput and pause time for web services
- **Equal Xms and Xmx** — prevents resize pauses under load
- **MaxGCPauseMillis=100** — G1 will try to keep GC pauses under 100ms

Monitor GC with:
```bash
java -Xlog:gc*:file=/var/log/app/gc.log:time,uptime:filecount=5,filesize=20m
```

## Results at Stackr

After applying these techniques to our provisioning API:
- P99 latency: 340ms → 28ms
- Throughput: 800 req/s → 4,200 req/s on the same hardware
- Database connection errors under load: eliminated

The biggest wins came from connection pool tuning and switching to projections — both are zero-cost changes that require no infrastructure.
