# PostgreSQL Internals: MVCC, WAL, and Why Your Queries Are Slow

Most developers treat PostgreSQL as a black box. You write SQL, rows come back. But understanding what happens inside is the difference between a 2ms query and a 2s query under load.

## MVCC: How Postgres Handles Concurrent Reads and Writes

PostgreSQL uses **Multi-Version Concurrency Control**. Instead of locking rows for reads, it keeps multiple versions of each row. Each transaction sees a snapshot of the database as it was when the transaction started.

Every row has two hidden system columns:

```sql
SELECT xmin, xmax, id, name FROM users LIMIT 5;

 xmin  | xmax  | id |   name
-------+-------+----+---------
  1234 |     0 |  1 | Eduardo
  1235 |  1238 |  2 | Alice
```

- `xmin` — the transaction ID that created this row version
- `xmax` — the transaction ID that deleted it (0 means it's still alive)

When you UPDATE a row, Postgres doesn't modify it in place. It marks the old row as dead (`xmax = current_txid`) and inserts a new row version. Old versions stick around until `VACUUM` cleans them up.

This is why:
- **Reads never block writes** — readers see old versions, writers create new ones
- **Writes never block reads** — same reason
- **VACUUM is important** — dead row versions accumulate, bloating tables and slowing down queries

## WAL: How Postgres Guarantees Durability

The **Write-Ahead Log** is how Postgres survives crashes. Before any change reaches the actual data files, it's written to the WAL — a sequential append-only log on disk.

On crash and restart:
1. Postgres reads the WAL from the last checkpoint
2. Re-applies any changes that were in WAL but not yet in data files
3. Rolls back any transactions that were in-progress at crash time

```sql
SHOW wal_level;
SHOW max_wal_senders;
```

WAL is also how **streaming replication** works. Replicas connect to the primary and stream WAL records in real time, replaying them to stay in sync. This is cheaper and more reliable than trigger-based replication.

## The Query Planner: Why EXPLAIN ANALYZE Is Your Best Friend

```sql
EXPLAIN ANALYZE
SELECT u.name, COUNT(o.id) as order_count
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2026-01-01'
GROUP BY u.id, u.name
ORDER BY order_count DESC
LIMIT 10;
```

```
Limit  (cost=1823.45..1823.48 rows=10 width=40) (actual time=12.3..12.3 rows=10 loops=1)
  -> Sort  (cost=1823.45..1848.45 rows=10000 width=40) (actual time=12.2..12.2 rows=10 loops=1)
       Sort Key: (count(o.id)) DESC
       -> Hash Join  (cost=420.00..1523.00 rows=10000 width=40) (actual time=3.1..10.5 rows=9500 loops=1)
             Hash Cond: (o.user_id = u.id)
             -> Seq Scan on orders  (cost=0.00..800.00 rows=50000 width=8)
             -> Hash  (cost=370.00..370.00 rows=4000 width=36)
                   -> Seq Scan on users  (cost=0.00..370.00 rows=4000 width=36)
                         Filter: (created_at > '2026-01-01')
```

The `Seq Scan on orders` is the problem. 50k rows scanned for a 10-row result — we need an index.

```sql
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_users_created_at ON users(created_at);
```

After adding indexes, the planner switches to `Index Scan` and the query drops from 12ms to 0.4ms.

## Common Performance Mistakes

**N+1 queries** — fetching a list, then querying each item:

```sql
-- BAD: 1 query for users + N queries for orders
SELECT * FROM users;
-- then for each user:
SELECT * FROM orders WHERE user_id = $1;

-- GOOD: 1 query with a JOIN
SELECT u.*, json_agg(o.*) as orders
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id;
```

**Missing partial indexes** — if you almost always filter by `status = 'pending'`:

```sql
CREATE INDEX idx_orders_pending ON orders(created_at)
WHERE status = 'pending';
```

This index is tiny and blazing fast compared to a full index on `(status, created_at)`.

**Autovacuum lag** — if writes are faster than autovacuum can clean up dead tuples, table bloat degrades scan performance. Monitor with:

```sql
SELECT schemaname, tablename, n_dead_tup, n_live_tup,
       round(n_dead_tup::numeric / nullif(n_live_tup, 0) * 100, 2) as dead_ratio
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```

If `dead_ratio` is above 10-20%, tune autovacuum or run `VACUUM ANALYZE` manually.

Understanding PostgreSQL at this level transforms how you design schemas, write queries, and debug production issues. The black box becomes predictable.
