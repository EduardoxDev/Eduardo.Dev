# Docker em produção: 7 erros que cometi e como evitar

Rodar Docker localmente é simples. Rodar em produção com clientes dependendo de disponibilidade é outra coisa.

Esses são erros reais que cometi na Stackr. Alguns custaram uptime. Todos me ensinaram algo.

## 1. Usar `latest` como tag em produção

```yaml
# Errado
image: myapp:latest

# Certo
image: myapp:a3f8b2c
```

`latest` é mutável. O mesmo deploy pode usar imagens diferentes dependendo de quando o pull aconteceu. Um bug entra silenciosamente porque "latest" mudou entre deploys.

Use o hash do commit como tag. Imutável, rastreável, reversível.

## 2. Rodar como root dentro do container

```dockerfile
# Errado — root por padrão
FROM node:20-alpine
COPY . .
CMD ["node", "server.js"]

# Certo
FROM node:20-alpine
RUN addgroup -S app && adduser -S app -G app
COPY --chown=app:app . .
USER app
CMD ["node", "server.js"]
```

Se um atacante explorar uma vulnerabilidade no app, dentro do container terá permissões de root. Com `USER app`, o dano é contido.

## 3. Não configurar limites de recursos

```yaml
services:
  api:
    image: myapp:a3f8c2b
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 256M
```

Sem limites, um container com memory leak pode consumir toda a RAM do host e derrubar outros serviços. Aprendi isso da forma difícil às 2h da manhã.

## 4. Imagens enormes porque não uso multi-stage build

```dockerfile
# Build stage — tem todas as ferramentas
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o app ./cmd/api

# Runtime stage — só o binário
FROM scratch
COPY --from=builder /app/app /app
ENTRYPOINT ["/app"]
```

Antes: imagem de 800MB. Depois: 8MB. Menos superfície de ataque, pull mais rápido, storage menor.

## 5. Healthcheck ausente

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD wget -qO- http://localhost:8080/health || exit 1
```

Sem healthcheck, o orquestrador considera o container saudável assim que sobe. Mas o processo pode estar vivo e a API morta — presa num loop de inicialização ou com conexão de banco perdida.

Com healthcheck, o tráfego só vai pro container depois que ele responde de verdade.

## 6. Logs indo para arquivos em vez de stdout

```go
// Errado — log para arquivo dentro do container
f, _ := os.OpenFile("/var/log/app.log", os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0644)
log.SetOutput(f)

// Certo — log para stdout
log.SetOutput(os.Stdout)
```

Containers são efêmeros. Logs em arquivos dentro do container são perdidos quando ele reinicia. Logs em stdout são capturados pelo Docker daemon e disponíveis via `docker logs` — e integram com qualquer sistema de observabilidade (Loki, ELK, CloudWatch).

## 7. Não ter graceful shutdown

```go
func main() {
    server := &http.Server{Addr: ":8080", Handler: router}

    go func() {
        if err := server.ListenAndServe(); err != http.ErrServerClosed {
            log.Fatal(err)
        }
    }()

    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGTERM, syscall.SIGINT)
    <-quit

    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    if err := server.Shutdown(ctx); err != nil {
        log.Fatal("Server forced shutdown:", err)
    }
}
```

Quando o Docker para um container, manda `SIGTERM`. Se o app não tratar, o Docker espera 10 segundos e manda `SIGKILL` — matando requests em andamento.

Com graceful shutdown, o app para de aceitar novas conexões e espera as existentes terminarem. Zero requests dropados.

---

Cada um desses erros virou uma checklist que rodo antes de qualquer deploy na Stackr. Containers em produção não são difíceis — mas exigem disciplina que o ambiente de desenvolvimento não força.
