# Redis além do cache: padrões que uso na Stackr em produção

Redis é frequentemente tratado como "só um cache". Na Stackr, é um dos componentes mais críticos da infraestrutura — e faz muito mais que cachear dados.

Aqui estão os padrões que uso em produção.

## Rate limiting com sliding window

O problema com rate limiting ingênuo (contador fixo por minuto) é o burst: 100 requests nos últimos 5 segundos de um minuto + 100 nos primeiros 5 do próximo = 200 requests em 10 segundos, mas ambas as janelas aceitam.

Sliding window resolve isso:

```go
var rateLimitScript = redis.NewScript(`
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])
local clearBefore = now - window

redis.call("ZREMRANGEBYSCORE", key, "-inf", clearBefore)
local count = redis.call("ZCARD", key)

if count >= limit then
    return 0
end

redis.call("ZADD", key, now, now .. math.random())
redis.call("EXPIRE", key, window / 1000)
return 1
`)

func (r *RateLimiter) Allow(ctx context.Context, key string) (bool, error) {
    now := time.Now().UnixMilli()
    result, err := rateLimitScript.Run(ctx, r.client,
        []string{"rl:" + key},
        now, r.windowMs, r.limit,
    ).Int()
    return result == 1, err
}
```

O sorted set usa timestamp como score. A janela desliza: antes de contar, remove tudo mais antigo que `window`. Sempre preciso e nunca permite burst.

## Distributed lock com Redlock

Para operações que só podem acontecer uma vez simultaneamente em múltiplos pods:

```go
type Lock struct {
    client *redis.Client
    key    string
    value  string
    ttl    time.Duration
}

func (r *RedisLocker) Acquire(ctx context.Context, resource string, ttl time.Duration) (*Lock, error) {
    value := uuid.New().String()
    key := "lock:" + resource

    ok, err := r.client.SetNX(ctx, key, value, ttl).Result()
    if err != nil || !ok {
        return nil, ErrLockNotAcquired
    }

    return &Lock{client: r.client, key: key, value: value, ttl: ttl}, nil
}

var releaseScript = redis.NewScript(`
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
`)

func (l *Lock) Release(ctx context.Context) error {
    _, err := releaseScript.Run(ctx, l.client, []string{l.key}, l.value).Int()
    return err
}
```

O script Lua no Release é crítico: verifica que o valor bate antes de deletar. Sem isso, um lock expirado pode ser deletado pelo processo errado.

```go
lock, err := locker.Acquire(ctx, "provision:"+serverID, 30*time.Second)
if err != nil {
    return ErrAlreadyProvisioning
}
defer lock.Release(ctx)

return provisionServer(ctx, serverID)
```

## Pub/Sub para eventos em tempo real

Na Stackr, usamos pub/sub para notificar o painel quando provisionamento termina:

```go
func (w *Worker) publishEvent(ctx context.Context, event ProvisionEvent) {
    data, _ := json.Marshal(event)
    w.redis.Publish(ctx, "provision:events:"+event.TenantID, data)
}

func (s *SubscriptionServer) handleWebSocket(conn *websocket.Conn, tenantID string) {
    sub := s.redis.Subscribe(ctx, "provision:events:"+tenantID)
    defer sub.Close()

    for msg := range sub.Channel() {
        var event ProvisionEvent
        json.Unmarshal([]byte(msg.Payload), &event)
        conn.WriteJSON(event)
    }
}
```

## Filas com BLPOP

Para processamento assíncrono sem overhead de um broker dedicado:

```go
func (w *Worker) processQueue(ctx context.Context) {
    for {
        result, err := w.redis.BLPop(ctx, 5*time.Second, "jobs:provision").Result()
        if err == redis.Nil {
            continue
        }
        if err != nil {
            w.logger.Error("queue error", "err", err)
            time.Sleep(time.Second)
            continue
        }

        var job ProvisionJob
        json.Unmarshal([]byte(result[1]), &job)
        w.processJob(ctx, job)
    }
}
```

`BLPOP` bloqueia até ter trabalho. Zero polling. Worker dorme de verdade quando não há nada para fazer.

## Uma coisa importante

Redis não é durável por padrão. Se o servidor cair sem AOF habilitado, você perde dados. Para rate limiting e locks temporários, tudo bem — recomeçam limpos. Para filas de jobs, habilite AOF ou considere Redis com persistência configurada.

```conf
appendonly yes
appendfsync everysec
```

Na Stackr, usamos Redis para tudo acima exceto dados financeiros, que vivem no PostgreSQL com durabilidade total. Certo lugar para cada ferramenta.
