# Construindo um motor de busca de código do zero

O [CodeSearch](https://github.com/EduardoxDev/CodeSearch-Style-Search-Engine) nasceu de uma frustração: ferramentas de busca de código genéricas não entendem contexto de programação.

Buscar `error handling` num motor comum retorna qualquer documento com essas palavras. Num buscador de código, você quer entender que `err != nil` é a forma idiomática em Go de fazer error handling, e rankear resultados com esse contexto.

## Índice invertido: a estrutura central

Um motor de busca não varre todos os documentos para cada query. Constrói um índice invertido: mapeamento de termo → lista de documentos que contém esse termo.

```
"goroutine" → [doc3, doc7, doc12, doc45]
"channel"   → [doc3, doc8, doc45, doc91]
"select"    → [doc3, doc12, doc91, doc102]
```

Query `goroutine channel select` → intersecção → `doc3`.

Em PostgreSQL, full-text search já implementa isso:

```sql
CREATE TABLE code_documents (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    repo        TEXT NOT NULL,
    filepath    TEXT NOT NULL,
    content     TEXT NOT NULL,
    language    TEXT,
    tsv_content TSVECTOR GENERATED ALWAYS AS (
        to_tsvector('english', content)
    ) STORED
);

CREATE INDEX idx_tsv ON code_documents USING GIN (tsv_content);
```

O `TSVECTOR` normaliza tokens automaticamente: `running`, `runs`, `ran` → `run`. O índice GIN torna a busca O(log n) em vez de O(n).

## Tokenização específica para código

Código tem tokens que texto normal não tem: `camelCase`, `snake_case`, `PascalType`. Um tokenizador genérico trata `handleRequest` como uma palavra opaca.

```go
var codeTokenRegex = regexp.MustCompile(
    `[a-z]+|[A-Z][a-z]*|[0-9]+|[a-zA-Z0-9]+_[a-zA-Z0-9_]+`,
)

func tokenizeCode(content string) []string {
    raw := codeTokenRegex.FindAllString(content, -1)
    tokens := make([]string, 0, len(raw))

    for _, t := range raw {
        lower := strings.ToLower(t)
        if len(lower) > 1 && !isStopWord(lower) {
            tokens = append(tokens, lower)
        }
    }
    return tokens
}
```

`handleRequest` → `handle`, `request`. Agora buscar `request` encontra `handleRequest`, `requestHandler`, `parseRequest` — todos relevantes.

## Ranking: TF-IDF + heurísticas de código

TF-IDF (Term Frequency × Inverse Document Frequency) é o baseline para ranking de relevância:

```go
func tfidf(tf float64, docsWithTerm, totalDocs int) float64 {
    idf := math.Log(float64(totalDocs) / float64(docsWithTerm+1))
    return tf * idf
}
```

Mas código tem heurísticas adicionais que importam:

```go
type CodeScorer struct {
    baseScore float64
    filepath  string
    language  string
}

func (s *CodeScorer) Score() float64 {
    score := s.baseScore

    // Arquivos de interface/API têm mais peso
    if strings.Contains(s.filepath, "interface") ||
       strings.Contains(s.filepath, "api") ||
       strings.Contains(s.filepath, "handler") {
        score *= 1.4
    }

    // Arquivos de teste têm menos peso
    if strings.HasSuffix(s.filepath, "_test.go") ||
       strings.Contains(s.filepath, "test/") {
        score *= 0.7
    }

    // Linguagem preferida do usuário
    if s.language == userPreferredLang {
        score *= 1.2
    }

    return score
}
```

## A query em SQL

```sql
SELECT
    d.id,
    d.repo,
    d.filepath,
    d.language,
    ts_rank(d.tsv_content, query) AS base_rank,
    ts_headline(d.content, query,
        'MaxWords=50, MinWords=20, StartSel=<mark>, StopSel=</mark>'
    ) AS snippet
FROM code_documents d,
     plainto_tsquery('english', $1) query
WHERE d.tsv_content @@ query
ORDER BY base_rank DESC
LIMIT 20;
```

`ts_headline` gera o snippet com o contexto do match — as linhas ao redor do termo encontrado, com destaque HTML embutido.

## O que aprendi sobre buscadores

Indexar é fácil. Rankear de forma útil é difícil.

O problema não é encontrar documentos que contêm o termo — é ordenar os 10.000 resultados para que o mais útil apareça primeiro. Cada heurística de ranking é uma hipótese sobre o que o usuário quer. E usuários querem coisas diferentes dependendo do contexto.

A próxima iteração do CodeSearch vai incluir feedback implícito: se o usuário clica no resultado 8 em vez do resultado 1, isso é sinal que o ranking está errado para aquela query. Aprender com esse feedback é o que separa um buscador bom de um ótimo.
